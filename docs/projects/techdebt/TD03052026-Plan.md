Convert the March 5 review findings into a hardening program that closes operational boundary seams with machine-verifiable proof.

## Cross-Doc Synthesis

Both March 5 documents converge on the same core program:
1. installation truth first
2. safe startup defaults with explicit dev mode
3. lifecycle ownership and cleanup symmetry
4. logging isolation and promotion-blocking proof gates

This plan adopts that shared structure and makes evidence and gate behavior normative.

## Locked Decisions

1. TD03052026 is boundary hardening work; no core architecture rewrite is in scope.
2. Documentation is never proof without a corresponding failing-then-passing gate.
3. Silent fallback is forbidden on critical runtime boundaries.
4. Runtime-safe defaults are mandatory on primary launcher paths.
5. Dev-permissive behavior must be explicit, named, and opt-in.
6. Provider lifecycle ownership is readiness-blocking.
7. Promotion-blocking gates require machine-readable artifacts.
8. Evidence is mandatory under `benchmarks/results/techdebt/td03052026/`.

## Execution Rules

1. All P0 seams are promotion-blocking and must close before new capability work on affected boundaries.
2. Every remediation slice must include:
   1. deterministic regression tests
   2. failing-then-passing proof for new or strengthened gates
   3. live behavioral verification at seam-specific repetition minimums
   4. mandatory evidence artifacts under `benchmarks/results/techdebt/td03052026/`
3. Critical runtime boundaries may only:
   1. fail fast with stable, documented behavior
   2. enter an explicit named dev or legacy mode with warning evidence
4. Silent fallbacks are forbidden for install, auth, lifecycle, provider ownership, and workspace isolation boundaries.
5. TD03052026 remains a hardening program, not an architecture rewrite.
6. Proof alone is not closure where ongoing regression risk exists; promotion-blocking gates must be enforced in CI, not only recorded as one-time evidence.

## Phase 0 Baseline and Scope Freeze (P0-prep)

Objective: freeze seam churn and establish measurable baseline proof before remediation.

Tasks:
1. Apply temporary feature freeze while P0 seams remain open.
2. During Phase 0 and open P0 workstreams, no feature changes may land that modify:
   1. install metadata or install docs
   2. server launcher defaults or reload semantics
   3. event subscription lifecycle
   4. provider client ownership or lifecycle
   5. workspace log routing
   unless the change belongs to an approved TD03052026 remediation slice.
3. Capture baseline operator commands in one tracked note.
4. Add hardening dashboard artifact with gate states.
5. Add baseline smoke CI minimum:
   1. fresh environment install from canonical docs command
   2. `python -c "import orket"`
   3. CLI or entrypoint help invocation
   4. API app construction smoke

Definition of done:
1. Baseline commands and pass/fail state are recorded.
2. Baseline smoke CI executes before WS-1 lands.

## Workstream Plan

## WS-1 Install Surface Convergence (P0)

Objective: one authoritative install path that always produces a runnable runtime.

Dependency authority rule:
1. The dependency authority must be singular. All install documentation, CI bootstrap, and developer setup flows must derive from the same canonical metadata source.

Tasks:
1. Select canonical dependency source.
2. Align README and quick-start to canonical source.
3. Remove or regenerate non-authoritative dependency artifacts.
4. Add install drift checker gate.

Definition of done:
1. Clean install from docs succeeds.
2. `python -c "import orket"` succeeds in clean env.
3. CI fails on install-docs drift.

Live verification minimum:
1. one clean install run.

## WS-2 Server Mode Hardening (P0)

Objective: safe startup defaults with explicit dev-only behavior.

Tasks:
1. Runtime-safe launcher defaults:
   1. host default `127.0.0.1`
   2. reload default `False`
2. Provide explicit named dev launcher or profile for reload workflows.
3. Define exactly which runtime inputs may override defaults.
4. No environment variable may silently enable dev semantics unless documented as dev-only.
5. Launcher override precedence must be documented and tested across CLI arguments, config values, and environment variables.

Definition of done:
1. Runtime launcher is local-only and non-reloading by default.
2. Dev-mode behavior is explicit and documented.
3. Tests assert default and opt-in behavior.

Live verification minimum:
1. one runtime launch validation run.

## WS-3 Event Subscriber Lifecycle Correctness (P0)

Objective: eliminate duplicate subscriber accumulation and prove teardown symmetry.

Lifecycle ownership invariant:
1. Every startup-side subscription registration has one and only one matching shutdown-side unsubscription for the same callback identity.

Tasks:
1. Enforce subscribe/unsubscribe symmetry in app lifespan.
2. Prevent duplicate registration of same callback identity where practical.
3. Add regression test that asserts subscriber count:
   1. before startup
   2. after startup
   3. after shutdown
   4. after repeated cycles

Definition of done:
1. No subscriber growth across repeated lifecycles.
2. Event delivery cardinality remains one-per-subscriber.

Live verification minimum:
1. at least 5 start/stop cycles.

## WS-4 Provider Client Close Contract (P0)

Objective: explicit provider transport ownership and deterministic shutdown.

Tasks:
1. Add explicit async close path for provider-owned clients.
2. Wire close ownership into runtime and API lifecycle.
3. Add repeated create/use/close tests.

Definition of done:
1. Provider clients are deterministically closed.
2. No connection-growth pattern in repeated cycles.

Live verification minimum:
1. at least 20 create/use/close cycles.

## WS-5 API Composition and Singleton Decoupling (P1)

Objective: reduce import-time mutable runtime singleton state.

Tasks:
1. Move runtime object ownership under app factory or app state.
2. Reduce import-time side effects in API surfaces.
3. Add multi-app isolation tests.
4. Anti-regression rule:
   1. `api.py` and related import surfaces must not construct mutable runtime singletons at import time for production app state.

Definition of done:
1. Multi-app instances remain isolated.
2. No production runtime singleton construction at import time.

Live verification minimum:
1. at least 2 distinct app or workspace contexts.

## WS-6 Workspace Logging Isolation (P1)

Objective: remove silent cross-run fallback behavior and define critical-path policy.

Critical logging paths covered:
1. run and event logs
2. crash and error logs
3. provider interaction logs where enabled
4. lifecycle and startup logs attributable to workspace or explicit dev/test mode

Allowed outcomes when required context is missing:
1. explicit fail-fast with stable behavior
2. explicit named dev or legacy fallback with warning event and mode marker
3. resulting fail-fast behavior or fallback markers must be stable and regression-testable (stable messages/codes or stable mode markers)

Tasks:
1. Enforce explicit workspace context on critical paths.
2. Restrict fallback behavior to explicit named modes only.
3. Add isolation tests for interleaved runs.

Definition of done:
1. No silent sink mixing across workspaces.
2. Fallback records are explicit and mode-tagged.

Live verification minimum:
1. at least 2 distinct run/workspace contexts.

## WS-7 Gate Enforcement and Auditability (P2)

Objective: convert policy claims into enforced, auditable CI gates.

Tasks:
1. Add CI-required gates for install parity, launcher defaults, lifecycle non-duplication, and provider close contract.
2. Add gate IDs and require all artifacts to reference IDs.
3. For each new promotion-blocking gate require:
   1. proof it fails against pre-fix behavior or seeded regression
   2. proof it passes post-fix
4. Update testing policy with gate commands and failure semantics.

Definition of done:
1. CI blocks merge on gate regression.
2. Gate artifacts and CI outputs are ID-addressable.

## Evidence Artifact Contract (Mandatory Per Workstream)

1. `commands.txt` with exact executed commands.
2. `environment.json` with OS, Python version, install mode, key env toggles.
3. `result.json` with pass/fail and named assertions.
4. Optional supporting logs:
   1. `stdout.log`
   2. `stderr.log`
   3. targeted runtime logs

## Live Verification Minimums

1. Startup/install checks: 1 clean run.
2. Lifecycle/subscriber checks: at least 5 start/stop cycles.
3. Provider lifecycle checks: at least 20 create/use/close cycles.
4. Logging isolation checks: at least 2 distinct run/workspace contexts.

## Gate State Definitions

1. Green:
   1. required tests pass
   2. live verification minimum is met
   3. mandatory artifacts are present
   4. no open severity-matching exception exists for that gate
2. Red:
   1. any required proof element is missing or failing
3. Waived:
   1. not permitted for P0 gates
   2. P1/P2 waiver requires recorded rationale and explicit expiry

## Risks and Change Control

1. Over-refactor risk:
   Keep changes boundary-focused; avoid architecture redesign during seam hardening.
2. False-confidence risk:
   No closure without failing-then-passing proof.
3. Feature-pressure risk:
   Hold feature work on affected seams until P0 gates pass.
4. Compatibility risk:
   Preserve permissive legacy behavior only as explicit named modes with warnings.

## Exit Criteria

1. All P0 workstreams are closed with mandatory artifacts and live verification minimums.
2. G1 through G4 are green.
3. Subscriber and provider ownership seams have passing regression coverage and live verification evidence meeting defined minimums, with no open linked defect for those failure classes.
4. Installation and startup defaults are safe and documented.
5. Promotion checklist artifact includes G1 through G7 with states, links, and timestamps.
