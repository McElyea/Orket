# Protocol-Governed Local Prompting Requirements (v1.2)

Last updated: 2026-03-05  
Status: Draft  
Owner: Orket Core

Reference:
1. `docs/projects/protocol-governed/requirements.md` (runtime governance authority)

## 1. Purpose

Define prompt/runtime compatibility rules for locally hosted models so protocol-governed execution remains deterministic, validator-enforced, and operationally debuggable across providers.
This provider-model profile contract serves as the Local Inference BIOS (nickname) for runtime behavior.

Core principle:
1. The model remains an untrusted proposal generator.
2. Prompting is an integration contract, not an authority boundary.
3. Runtime validators and schemas remain the enforcement authority.

## 2. Scope

In scope:
1. Provider-model prompt-template compatibility.
2. Structured output and tool-call contract hardening.
3. Runtime sampling, stop, and history policies for deterministic paths.
4. Validator-driven repair loops and anti-meta enforcement surfaces.
5. Measurable conformance and upgrade drift gates.

Out of scope:
1. Installer/wizard UX for LM Studio or Ollama lifecycle management.
2. General model quality benchmarking beyond protocol-compliance surfaces.

## 3. Definitions

1. Prompt profile: Provider+model scoped rendering and sampling contract.
2. Template fidelity: Rendered prompt preserves native training markers.
3. Output contract: Exact machine-parseable schema required by runtime.
4. Repair loop: Deterministic retry path driven by validator error codes.
5. Protocol chatter: Meta or conversational text outside required payload.
6. Task class: One of `strict_json`, `tool_call`, `concise_text`, `reasoning`.
7. Profile ID: Stable identifier for a resolved provider-model contract.
8. Local Inference BIOS: Informal name for the provider-model profile contract and resolution path.

## 4. Hard Requirements

### LP-01: Provider-Model Prompt Profiles (MUST)

Runtime MUST resolve prompt behavior by `(provider, model)` rather than model-only.

Minimum profile fields:
1. `profile_id` (string)
2. `template_family` (`openai_messages`, `chatml`, `inst`, `custom`)
3. `template_variant` (string; for example `llama3_header`, `llama2_inst`, `deepseek_think`)
4. `template_source` (`runtime_metadata`, `profile_override`, `provider_builtin`, `unknown`)
5. `template_version` (string; allow `unknown` or `none`)
6. `allowed_roles` (array of strings)
7. `system_prompt_mode` (`native`, `user_injection`)
8. `context_budget_tokens` (integer > 0)
9. `history_policy` (string; see LP-07)
10. `stop_sequences_by_task_class` (map `task_class -> [stop...]`)
11. `supports_assistant_prefill` (capability)
12. `prefill_strategy` (`none`, `assistant_prefix`, `forced_token`, `provider_specific`)
13. `tool_call_mode` (`native`, `json_wrapper`)
14. `tool_contract` object:
   - `tool_manifest_injection` (`system`, `user`, `none`)
   - `tool_call_schema` (schema identifier string)
   - `tool_result_role` (`tool`, `user`, `assistant`)
15. `allows_thinking_blocks` (boolean)
16. `thinking_block_format` (`none`, `xml_think_tags`, `provider_native`)
17. `sampling_bundles` (map by task class; see LP-08)

LP-01 visualization (profile resolution path):

```text
[provider, model, task_class]
          |
          v
[profile registry lookup] ----miss----> [fallback policy (LP-14)]
          |
          v
[resolved profile_id + contract fields]
          |
          v
[adapter contract check: roles/template/stop/sampling]
          |
          v
[generation call]
```

### LP-02: Native Template Fidelity and Stable Render Hashing (MUST)

Rendered prompts MUST use model-native turn markers or an exact equivalent generated by runtime metadata/template machinery.

Requirements:
1. Do not mix incompatible marker families.
2. Preserve delimiter and newline placement required by tokenizer/template behavior.
3. Rendered prompt hashing MUST define canonicalization rules:
   - normalize line endings to `\n`,
   - strip trailing whitespace,
   - no Unicode normalization unless explicitly enabled.
4. Hashed material MUST be the exact byte sequence submitted to provider model input after template render and LP-02 canonicalization.
5. `template_version` MUST be recorded alongside `template_hash`.
6. `template_hash_alg` MUST use a fixed versioned algorithm identifier (default `sha256`) and MUST be recorded with every `template_hash`.

### LP-03: Role Contract (MUST)

When runtime enforces role names, messages MUST use supported role names only.

Requirements:
1. Allowed roles MUST be declared in the prompt profile (`allowed_roles`) and MUST match the provider adapter's supported role set for that runtime path.
2. If a runtime path requires a `tool` role, `tool` MUST be included in `allowed_roles`.
3. Do not invent custom roles in outbound payloads.
4. For profiles using `system_prompt_mode=user_injection`, system content MUST be injected into the first user turn using a deterministic wrapper, and outbound payload MUST omit the `system` role entirely.

### LP-04: Stop Sequence Contract (MUST)

Each profile MUST define stop behavior per task class.

Requirements:
1. Stop sequences MUST be applied in the runtime call where supported.
2. Structured-output task classes MUST include an explicit sentinel stop option.
3. Compliance tests MUST verify stop behavior per `(profile, task_class)` and MUST include at least one structured-output test where sentinel stop is the terminating condition.
4. If multiple stop sequences are configured, earliest encountered stop MUST terminate generation (provider behavior permitting).
5. Conformance MUST include a `must-stop` case where generation would continue indefinitely unless sentinel stop is honored.
6. Stop binder MUST order stops deterministically as `[sentinel_stops...] + [template_eot_stops...] + [provider_default_eos...]` unless profile explicitly overrides ordering.

### LP-05: Schema-First Output Contract (MUST for parseable outputs)

For `strict_json` and `tool_call` task classes, prompt assembly MUST include:
1. exact schema or fixed-key contract,
2. explicit payload-only output constraint,
3. no-markdown/no-prose constraint.
4. for `tool_call`, serialization and role behavior MUST follow `tool_contract` in LP-01.

### LP-06: Assistant Prefill Policy (SHOULD; MUST when configured)

Assistant prefilling SHOULD be used for strict structured outputs when safe.

Requirements:
1. Prefill MUST be applied when `prefill_strategy != none`.
2. Capability alone is insufficient; `supports_assistant_prefill=true` does not imply mandatory prefill.
3. If prefill is unsupported or unsafe for a profile, `prefill_strategy=none` and stricter LP-09 repair-loop enforcement is required.

### LP-07: Context Minimization and History Policy (MUST)

Prompt context MUST be bounded and policy-driven.

Requirements:
1. Support bounded transcript windows.
2. Support seat-only history mode.
3. Default history policy MUST be explicitly named as `bounded_head_tail` with:
   - fixed token budget,
   - deterministic trimming order,
   - deterministic tie-break ordering.
4. Active context trimming MUST honor `context_budget_tokens` before provider submission.
5. `context_budget_tokens` MUST be computed with the tokenizer used by the resolved provider-model profile (or a declared equivalent), and token counter source MUST be recorded in conformance artifacts.
6. Strict tool paths MUST exclude non-essential context/protocol chatter.
7. `history_policy` MUST be selected from a versioned registry:
   - `bounded_head_tail_v1`
   - `seat_only_bounded_head_tail_v1`
   - `none_v1`
8. `bounded_head_tail` is an alias for `bounded_head_tail_v1`.

LP-07 visualization (bounded head-tail):

```text
full history: [HEAD................MIDDLE................TAIL]
keep:         [HEAD........] + [........TAIL]
drop:                        [MIDDLE]
rules: deterministic token budget + deterministic trim order
```

### LP-08: Sampling Bundles (MUST)

Each profile MUST define a sampling bundle for every required task class.

Required classes:
1. `strict_json`
2. `tool_call`
3. `concise_text`
4. `reasoning`

Each sampling bundle MUST include:
1. `temperature`
2. `top_p`
3. `top_k`
4. `repeat_penalty`
5. `max_output_tokens`
6. `seed_policy` (`fixed`, `none`, `provider_default`)
7. `seed_value` (required when `seed_policy=fixed`)

Seed policy semantics:
1. `seed_policy=provider_default` means runtime MUST omit explicit seed parameters.
2. Conformance artifacts MUST record whether provider behavior is deterministic-by-default for the tested profile.
3. Conformance artifacts MUST record per-field behavior for `temperature`, `top_p`, `top_k`, `repeat_penalty`, `max_output_tokens`, and `seed` (`honored`, `ignored`, `clamped`, or `approximated`).

Guardrails:
1. `strict_json` and `tool_call` MUST use bounded `max_output_tokens`; unbounded output is forbidden.
2. `strict_json` repeat penalty MUST remain within a profile-approved safe range demonstrated by conformance (default max `1.05`); values above `1.05` require explicit profile exception with conformance evidence.
3. DeepSeek-R1-style reasoning profiles SHOULD use nonzero temperature in `0.5-0.7` (target `0.6`) unless profile declares a different validated range.

### LP-09: Deterministic Repair Loop (MUST)

Invalid structured output MUST trigger a bounded validator-driven corrective reprompt loop.

Requirements:
1. Validator MUST emit stable `error_code` values from a fixed registry.
2. Minimum registry families:
   - `JSON_PARSE_ERROR`
   - `SCHEMA_MISMATCH`
   - `EXTRANEOUS_TEXT`
   - `TOOL_SHAPE_INVALID`
   - `STOP_NOT_HONORED`
3. Corrective reprompt MUST be derived deterministically from:
   - `error_code`
   - `short_error_detail`
   - `prior_output_excerpt_hash`
4. Retry limits and terminal failure behavior MUST be deterministic.

LP-09 visualization (deterministic repair loop):

```text
[model output] -> [validator]
      |              |
      | valid        | invalid(error_code)
      v              v
   [accept]   [deterministic reprompt builder]
                     |
                     v
              [retry attempt n+1]
                     |
                 [bounded max]
                     |
          [terminal deterministic failure]
```

### LP-10: Anti-Meta Guardrails (MUST in strict modes)

Strict mode validation MUST include:
1. `only_contract_payload` check (no non-whitespace outside payload),
2. markdown fence detection,
3. intro-phrase detector using a configurable denylist.

Requirements:
1. Denylist MUST be profile-scoped.
2. Payload-only validation MUST allow only ASCII whitespace outside payload (`0x09`, `0x0A`, `0x0D`, `0x20`) and MUST trim at most once before validation.
3. If `allows_thinking_blocks=false`, any thinking markers MUST fail validation.
4. If `allows_thinking_blocks=true`, thinking blocks are permitted only before strict payload begins and MUST NOT appear after payload start.
5. For permitted thinking blocks, adapter MUST exclude them from strict payload/tool parsing so thought text cannot be interpreted as tool-call payload.
6. If provider exposes native thinking suppression controls, strict task classes SHOULD prefer native suppression before manual block filtering.
7. If strict-mode response contains thinking blocks, validation MUST fail unless remaining non-thinking payload fully satisfies strict contract.
8. Thinking content MUST be ignored for payload parsing but treated as extraneous text if it appears outside permitted locations for the active profile.
9. Violations MUST fail validation and route to LP-09 repair flow.

### LP-11: Rendered-Input Observability (MUST)

Runtime MUST capture enough metadata to explain regressions without storing sensitive prompt content.

Minimum telemetry fields:
1. `provider`
2. `model`
3. `profile_id`
4. `task_class`
5. `template_family`
6. `template_version`
7. `template_hash`
8. `template_hash_alg`
9. `rendered_prompt_byte_count`
10. `sampling_bundle`
11. `stop_sequences_by_task_class`
12. `effective_stop_sequences`
13. `history_policy`

Requirements:
1. Validator inputs MUST include `(profile_id, task_class)`.

### LP-12: Runtime Compatibility Matrix Coverage (MUST)

Active providers (`ollama`, `lm_studio`) MUST have conformance tests for each active model profile.

Coverage must prove:
1. template render-path correctness,
2. role-set compliance,
3. stop behavior by task class,
4. structured output compliance,
5. validator and repair-loop behavior.

### LP-13: Upgrade Drift Gates (MUST)

Provider/runtime/model upgrades MUST run conformance suites and capture profile deltas before promotion.

Minimum artifacts:
1. profile compatibility report,
2. error-family failure ledger,
3. before/after compliance metrics.

### LP-14: Fallback Semantics (MUST)

When profile resolution fails:
1. strict paths MUST fail closed, or
2. runtime MUST explicitly downgrade to a documented safe profile and emit compatibility warnings.

Silent fallback is forbidden.

### LP-15: Render Verification Surface (MUST for promoted providers; SHOULD otherwise)

The adapter MUST provide at least one render verification path:
1. provider-native rendered prompt preview endpoint, or
2. deterministic local renderer matching provider template behavior byte-for-byte, or
3. captured final prompt string hash returned/derivable from provider response metadata.

Conformance tests MUST compare expected vs observed render hash where verification is available.
Compared hash MUST be the post-canonicalization rendered prompt hash defined by LP-02 rules.

### LP-16: Template Integrity and Audit (MUST)

When `template_family != openai_messages`, runtime MUST audit template content for suspicious conditional logic and hidden trigger behavior before profile promotion.

Requirements:
1. Audit scope includes tokenizer chat templates and profile override templates.
2. Suspicious conditional triggers (for example message-content-based hidden branches) MUST fail closed until explicitly whitelisted.
3. Audit MUST detect at minimum:
   - conditional branches based on message content,
   - hidden role remapping,
   - insertion of undeclared tool instructions,
   - string/regex trigger paths activating alternate render behavior.
4. Whitelisting requires traceable approval record and conformance re-run artifacts.
5. Audit MUST emit a structured report artifact including detected constructs, pass/fail decision, and whitelist references used.

This matrix defines baseline expectations using explicit `template_family` semantics.

| LM Studio | `template_family=openai_messages`; `template_variant=lmstudio_openai_chat`; `template_source=provider_builtin`; profile override allowed via explicit profile exception. | `template_family=openai_messages`; `template_variant=lmstudio_chatml_metadata`; `template_source=runtime_metadata`; override only by explicit profile exception. | `template_family=openai_messages`; `template_variant=lmstudio_mistral_chat`; `template_source=provider_builtin`; legacy markers handled via profile override. | `template_family=custom`; `template_variant=deepseek_think_override`; `template_source=profile_override`; thought blocks excluded from strict parsing. |

## 6. Prompt Assembly Policy

Required ordering for strict structured tasks:
1. global contract constraints,
2. task payload,
3. exact output schema,
4. optional single exemplar,
5. final payload-only constraint immediately before generation.

Rules:
1. In strict structured tasks, schema MUST appear after task payload to reduce schema/task mismatch.
2. Final line MUST restate payload-only output immediately before generation to leverage recency bias.
3. Zero-shot is default.
4. At most one exemplar is allowed by default; multi-example few-shot packs are disallowed in strict tool mode.

## 7. Default Sampling Policy

Baseline profile ranges:
1. `strict_json`: `temperature 0.0-0.2`, conservative `top_p/top_k`, `repeat_penalty <= 1.05`.
2. `tool_call`: near-deterministic temperature with constrained decoding and bounded output tokens.
3. `concise_text`: low-mid temperature with bounded output tokens.
4. `reasoning`: moderate temperature for designated reasoning profiles only (`0.5-0.7` for DeepSeek-R1-style profiles).

Seed policy:
1. Deterministic runs SHOULD use `seed_policy=fixed` where provider supports it.
2. If provider ignores seed, runtime MUST record this in conformance artifacts.

## 8. Validation and Quality Gates

### 8.1 Required metrics

1. JSON parse pass rate (`strict_json`): target `>= 98%` on 1000-case suite.
2. Tool-call shape correctness: target `>= 99%` on 500-case suite.
3. Protocol-chatter rate in strict modes: target `< 2%`.
4. Deterministic repair-loop convergence: bounded retries with stable terminal error code on failure.

### 8.2 Failure handling

On regression:
1. block profile promotion,
2. emit failing profile/runtime/task-class path,
3. emit top error code families,
4. require profile fix or explicit rollback decision.

## 9. Acceptance Criteria

A profile is promotion-eligible only when:
1. LP-01 through LP-16 are satisfied.
2. Section 8 metrics meet thresholds for that profile.
3. Live run evidence is captured on the real runtime path (primary vs fallback).
4. Artifacts include render metadata, validator outcomes, and repair-loop outcomes.

## 10. Requirement Realization Slices

1. Add prompt profile registry schema and validation.
2. Add adapter role-set enforcement and profile role compatibility checks.
3. Add template canonicalization and render-hash utility library.
4. Add stop-sequence binder per `(profile, task_class)`.
5. Add strict output-contract builder and prefill policy router.
6. Add stable error-code registry and deterministic reprompt builder.
7. Add validator-driven repair-loop service with bounded retries.
8. Add conformance harness for Ollama and LM Studio.
9. Add upgrade drift report artifacts and promotion gate checks.
10. Add template integrity audit and whitelist gate for non-`openai_messages` profiles.

## 11. Operator Checklist

Before enabling or updating a profile:
1. verify provider/model mapping resolves expected profile,
2. verify rendered marker family and role set match profile contract,
3. run strict JSON and tool-call conformance suites,
4. run live script sweep (excluding long-running quant sweeps when intentionally deferred),
5. publish compliance artifacts and review failure ledger by error family.
6. for non-`openai_messages` profiles, verify template integrity audit and whitelist status before promotion.
