Reliability and Correctness

Fault injection test suite — systematically inject asyncio.TimeoutError, httpx.ConnectError, and aiosqlite lock timeouts into the hot path and verify the system transitions cards to blocked rather than hanging or corrupting state. This is your behavioral review in test form.
State reconciliation proof — implement W2-F from the plan and run it against a real Gitea instance + local SQLite after a deliberate divergence. Publish the reconciliation output as a live proof.
Lease contention stress test — run two concurrent card dispatchers targeting the same card and verify only one wins the lease and the other fails gracefully. Proves the concurrency model is real.
Tool gate bypass detection — construct a card that attempts to write outside the workspace root, verify the gate rejects it, and log the policy violation. Then verify the same attempt on the non-governed path is also caught (it currently isn't — this surfaces the gap).
Truncation recovery falsification test — deliberately feed the tool parser a truncated two-tool response (write + status update) and verify the partial execution is detected and surfaced, not silently completed. This proves W1-A actually works after you implement it.

Codex review of these ideas

Ship-Risk Debt

End-to-end card lifecycle proof: strongest “prove Orket works” idea. Do it after Wave 1 blockers, with a tiny real task and durable artifacts staged before any publication.
Fault injection test suite: excellent, high-priority. It directly targets timeout, connection, and lock failure behavior that can corrupt truth or hang execution.
State reconciliation proof: good, but depends on W2-F. First define the authority policy: SQLite wins, Gitea wins, or halt-and-alert.
Lease contention stress test: valuable W1-C/W2-E acceptance proof. Best after lease fixes so it proves graceful failure instead of rediscovering a known gap.
Tool gate bypass detection: high-value, but treat the non-governed path claim as a bug to fix if confirmed, not just a detector. Keep the unsafe path attempt inside a safe temp fixture.
Truncation recovery falsification test: highest-signal Wave 1 test. It directly proves W1-A closes the silent partial-execution failure mode.
Gitea-native card creation flow: product-useful but too early. Do webhook schema validation, reconciliation, and lease authority first.
Multi-card parallel execution proof: valuable after lease/read-write lock work. Before that, it is likely to surface already-known contention.

Exploration-Safe Debt

Multi-model role matrix run: interesting benchmark experiment. Keep it subordinate to real card lifecycle proof; ODR evidence should not substitute for runtime acceptance.
Companion provider runtime matrix expansion: useful for model selection, but expensive and publication-gated. Stage first, publish only after explicit approval.
Web UI dashboard: useful debugging surface, but defer until the CLI inspector exists. Keep it read-only to avoid creating new authority.
CLI card inspector: best “New Surfaces” item. Low-risk, high debugging value. If it adds python main.py --inspect, update authority docs in the same change.
Protocol ledger reader/viewer: strong bounded observability work. It makes existing evidence inspectable without adding runtime authority.
Turn-level trace export: useful, but make it an explicit projection with redaction, not a second source of truth.
Adversarial prompt corpus expansion: good hardening after W1-A/tool-gate cleanup. Report exact compare scope and pass criteria.

Self-Deception Debt

Determinism campaign over N runs: good idea, but “identical tool calls” is too strong for live local models unless seed/provider/runtime controls are explicit. Use fake-model identity proof plus live bounded replay/verdict proof.
Publish a reproducibility contract: valuable but dangerous as a product claim. Start as lab/proof-tier under ORKET_DETERMINISM_GATE_POLICY.md.
Model-agnostic integration test: excellent regression baseline, and the referenced fake adapter exists. Do not label it live proof.
CURRENT_AUTHORITY.md audit against live code: good governance direction, but check_docs_project_hygiene.py alone is project hygiene, not full authority-vs-code proof. It needs a dedicated checker to support that claim.
