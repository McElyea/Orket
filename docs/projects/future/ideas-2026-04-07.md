Here's what I'd build around the Card Viewer/Runner as the anchor:
Tab 1 — Cards (your starting point)
The main work surface. Left sidebar lists cards filterable by state (queued, running, failed, done), type, and tier. Right panel is the card detail — prompt, last run output, run history, and a big "Run" button. Live streaming output when a card is running. This is the "cockpit" — the thing you look at most.
Tab 2 — Run History / Ledger
A chronological feed of every run: card ID, model, duration, pass/fail, cost, determinism hash. Filterable. Click a row to expand full output, artifacts, and the canonical hash comparison. This surfaces what's currently buried in benchmarks/results/ JSON files.
Tab 3 — Benchmark / Quant
The sweep and scoring view. Run a quant sweep, watch it progress, see pass/fail by tier, score distribution, and regressions highlighted in red. Right now this is entirely CLI and output files — a UI here would be genuinely transformative for your workflow.
Tab 4 — Model Health Matrix
Provider/model health matrix. Connect to Ollama or LMStudio, run the provider runtime matrix, see which rig classes pass per model. Toggle a model active/inactive. This is a natural UI because the matrix output already has a table shape.
Tab 5 — System / Settings
Gitea connection status, sandbox lifecycle health, spool status (how many events are backed up), active API keys, and the control plane convergence status summary. One place to see if anything is on fire.
The key design principle I'd push for: every tab should have a status indicator in the tab itself (a green dot, a count badge, a spinner). Cards shouldn't just be a list — tabs 2 and 3 should tell you something is wrong before you even click them.
