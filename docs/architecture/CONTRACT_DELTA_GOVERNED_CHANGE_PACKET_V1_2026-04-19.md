# Contract Delta Proposal

## Summary
- Change title: Governed Change Packet v1 trusted-kernel, packet, and standalone-verifier admission
- Owner: Orket Core
- Date: 2026-04-19
- Affected contract(s): `docs/specs/FIRST_USEFUL_WORKFLOW_SLICE_V1.md`, `docs/specs/TRUST_REASON_AND_EXTERNAL_ADOPTION_V1.md`, `docs/specs/GOVERNED_CHANGE_PACKET_TRUSTED_KERNEL_V1.md`, `docs/specs/GOVERNED_CHANGE_PACKET_V1.md`, `docs/specs/GOVERNED_CHANGE_PACKET_STANDALONE_VERIFIER_V1.md`

## Delta
- Current behavior: The externally admitted repo-change slice shipped a witness bundle, campaign report, and offline verifier report, but it had no first-class governed packet surface, no standalone packet verifier, and no separate trusted-kernel contract for the first outside-operator packet.
- Proposed behavior: Admit one bounded `governed_change_packet.v1` surface for `trusted_repo_config_change_v1`, add a bounded trusted-kernel contract plus machine-checkable model output, add an inspection-only standalone packet verifier with required-role, authority-ref, stable-digest, and claim-downgrade diagnostics, and update the external trust boundary so the packet is the primary operator entry artifact while the underlying authority artifacts remain claim-bearing proof authority.
- Why this break is required now: The governed-change-packet lane is explicitly about turning the already admitted repo-change proof slice into an independently inspectable outside-operator packet without broadening the external trust boundary beyond what the existing evidence supports.

## Migration Plan
1. Compatibility window: Existing trusted repo change commands, witness bundle outputs, campaign report outputs, and offline verifier outputs remain canonical and continue to be generated by the new packet path.
2. Migration steps:
   1. Add the new trusted-kernel, packet, and standalone-verifier specs.
   2. Add the packet generation, standalone verification, trusted-kernel model, and adversarial benchmark scripts.
   3. Add verifier diagnostics that explain missing roles, authority-ref load or stable-digest drift, and offline claim-ladder downgrades without creating new verdict values.
   4. Update the useful-workflow and trust-reason specs to admit the packet as the primary operator entry artifact.
3. Validation gates:
   1. `python -m pytest -q tests/scripts/test_governed_change_packet.py tests/scripts/test_trusted_run_proof_foundation.py tests/scripts/test_first_useful_workflow_slice.py`
   2. `python scripts/proof/run_governed_repo_change_packet.py`
   3. `python scripts/proof/verify_trusted_run_proof_foundation.py`

## Rollback Plan
1. Rollback trigger: Packet or standalone-verifier contradictions that invalidate the admitted repo-change proof slice.
2. Rollback steps:
   1. Remove the governed packet specs and scripts from current authority.
   2. Restore evaluator guidance to the witness-bundle plus offline-verifier path only.
   3. Regenerate the trusted repo change proof artifacts on the pre-packet authority path.
3. Data/state recovery notes: Packet outputs and the staged adversarial benchmark are derived artifacts only; they can be regenerated after rollback.

## Versioning Decision
- Version bump type: minor contract extension within the existing admitted repo-change slice
- Effective version/date: 2026-04-19
- Downstream impact: Outside-operator guidance now points first to the governed packet and standalone verifier, verifier output is more explicit about required roles, authority refs, and claim downgrades, but the underlying witness bundle, campaign report, and offline verifier remain the underlying proof authority and public trust wording stays capped at `verdict_deterministic` for `trusted_repo_config_change_v1`.
